Frontend Architecture & Core Logic1. Domain & TerminologyDomain: Real Estate, Chalet Rentals, Heavy Machinery Leasing.Tenant: The agency or property owner (e.g., smar).Core Action: "Schedule a Viewing", "Book a Chalet", or "Submit Leasing Application" (NOT "Add to Cart").2. The 4-Layer Frontend ArchitectureStrictly separate UI from Business Logic:@data: API calls using publicApi and adminApi Axios instances.@domain: Data transformation, logic hooks (e.g., useListings(), useBookingSubmit()).@presentation/components: Generic, tenant-agnostic UI (Buttons, DatePickers).@presentation/pages/[slug]: Tenant-specific layouts and logic (e.g., src/pages/smar/).3. Dual-Mode Routing EngineEvery tenant must support a 2D and 3D/Spatial architectural split.TenantPages.jsx is the single source of truth. Map slugs to components here (e.g., smar.home, smar.chalets).Normal UI (normal/): Standard grid views, forms, and booking calendars.Spatial UI (spatial/): 2.5D Parallax scenes, Kinetic Galleries, Cinematic Videos.4. State ManagementUse Context or Zustand for the Booking Session (Dates, Guests, Chalet ID).Forms must handle complex validations (Check-in/Check-out dates, preventing double bookings).
