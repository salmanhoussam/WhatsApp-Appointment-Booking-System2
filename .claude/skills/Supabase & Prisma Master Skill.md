user-invocable: trueSupabase & Prisma Mastery DirectivesWhen interacting with the database, scaffolding features, or handling file uploads, you MUST adhere to this exact architecture.1. Entity Dictionary (Naming Conventions)The client and frontend may use different terminology than the database. Here is the strict translation map:"Property" / "Chalet" / "Villa" ➔ Prisma Model: Unit"Order" / "Reservation" ➔ Prisma Model: Booking"Customer" / "Guest" ➔ Prisma Model: User (where role = "customer" or implied by Booking relationship)"Agency" / "Tenant" ➔ Prisma Model: Client (Identified by clientSlug)2. Image & File Storage ArchitectureNEVER save base64 images to the database. NEVER save files to the local file system.The Upload: When an Admin uploads a Unit image, it MUST be uploaded to the Supabase Storage Bucket (e.g., properties bucket).The Path: The storage path must be tenant-isolated: /[clientSlug]/units/[unit_id]/[filename.jpg].The Save: Only the Public URL of the uploaded image is saved to the Prisma database in the images array field of the Unit model.3. The Golden Rule of Database SyncingIf you EVER add a new column (like payment_method) or table to schema.prisma:You MUST inform the user that the Python backend code and the actual PostgreSQL database are now out of sync.You MUST instruct the user to run exactly these two commands in their backend terminal:prisma generate (To rebuild the Python client)prisma db push (To create the missing columns/tables in Supabase PostgreSQL)If they do not run this, the FastAPI server will crash with EngineConnectionError or DataError.
