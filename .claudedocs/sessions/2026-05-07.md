## الهدف

بناء صفحات الواجهة العامة (Public Pages) كـ Generic Pages قابلة لإعادة الاستخدام مع أي tenant، تعمل حسب `module_key` المشتق تلقائياً من `active_services` الخاصة بكل مستأجر.

الصفحات المبنية:
- **CatalogPage** — عرض القائمة / المنتجات بحسب module
- **CartPage** — سلة الطلبات + Checkout
- **ReservePage** — نموذج حجز موعد

## القرارات الهندسية

- **Session ID للمتاجر:** يُنشأ عبر `crypto.randomUUID()` ويُخزن في localStorage بـ key `{slug}_cart_session` — عزل كامل بين الـ tenants.
- **Module key priority:** restaurant > store > catalog — أول `active_service` مطابق يحدد المسار.
- **Cart key contract:** `catalogItemId` داخلياً في Zustand (convention المطاعم)؛ يُحوَّل إلى `catalog_item_id` في جميع payloads الـ API (catalog-contract Phase 54).
- **Store checkout flow (2 steps):** sync local cart → `POST /store/cart` لكل عنصر ← ثم → `POST /store/orders` بـ session_id. السبب: الـ API يقرأ الكارت من DB وليس من body الطلب مباشرة.
- **Template modification:** أضيف `onAddToCart` فقط، لا تعديل على logic العرض أو التخطيط — التوافق الخلفي محفوظ.
- **FM12 Safety:** لا `useScroll` أو `useTransform` في أي من الصفحات الجديدة — فقط `animate={}`, `whileTap` → آمن بدون lazy loading.

## الخطة القادمة — Phase 58: Route Registration

### الهدف
تسجيل مسارات الصفحات العامة الجديدة في ملفات الراوتر الخاصة بكل tenant، وربط `CatalogPage / CartPage / ReservePage` بالـ Router.

### المهام

1. **تعديل `caracas.routes.jsx`**
   - إضافة: `/{slug}/menu` → `CatalogPage` (moduleKey = restaurant)
   - إضافة: `/{slug}/cart` → `CartPage`
   - إضافة: `/{slug}/reserve` → `ReservePage`

2. **تعديل `footlab.routes.jsx`**
   - إضافة: `/{slug}/store` → `CatalogPage` (moduleKey = store)
   - إضافة: `/{slug}/cart` → `CartPage`

3. **مراجعة `smar.routes.jsx`** (اختياري)
   - إذا كان smar سيضيف catalog/reserve مستقبلاً.

4. **(اختياري) TrialTenantResolver**
   - توجيه dynamic routes بناءً على `active_services` — tenant جديد يفتح `/catalog` تلقائياً إذا كان عنده `catalog` service.

5. **اختبار end-to-end**
   - فتح كل مسار والتأكد من ظهور الصفحة الصحيحة مع بيانات حقيقية.
   - التحقق أن `ReservePage` لا تظهر إلا إذا كانت `reservations` في `active_services`.

### التحديات المتوقعة

| التحدي | الحل |
|--------|-------|
| `ReservePage` تظهر لـ tenants بدون `reservations` | الصفحة تفحص `hasReservations` داخلياً وتعرض رسالة "غير متاح" — الـ route يمكن إخفاؤه أيضاً من الـ nav عبر `getNavItems()` |
| `primary_color` لا يُطبَّق على الـ nav | `TenantModuleNav` يقرأ `config.primary_color` تلقائياً من `useTenantConfig` |
| Glassmorphism غير متطابق | جميع الصفحات تستخدم `rgba(255,255,255,0.04)` + `backdrop-blur` — متوافق مع GS MAR |
