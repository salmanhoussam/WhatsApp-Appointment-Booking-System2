### 🟡 النقطة 1 — Base Price Fallback (إصلاح منطق السعر)

**المشكلة:**
إذا لم يُسعَّر يوم معين في التقويم (`Price` table)، يعود النظام بـ 0 بدلاً من سعر الوحدة الافتراضي (`Unit.price`).

**الملفات المؤثرة:**
- `app/services/price_service.py` — منطق حساب السعر
- `app/repositories/price_repo.py` — جلب الأسعار من DB

**الحل المقترح:**
في `get_price_for_range()` أو ما يعادلها:
```python
# إذا ما وُجد سعر في التقويم → استخدم Unit.price كـ fallback
daily_price = calendar_price or unit.price or 0
```

**الأثر:** منخفض المخاطر، إصلاح سطر أو سطرين فقط.

**المشكلة:**
لا يوجد جدول `GalleryImage` في قاعدة البيانات. الصور تُرفع مباشرة إلى Supabase Storage لكن:
- لا يمكن ترتيبها (sort_order)
- لا يمكن إخفاء صورة دون حذفها
- لا يمكن إضافة caption عربي/إنجليزي

**المقترح من التقرير:**
```prisma
model GalleryImage {
  id         String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  clientId   String   @map("client_id") @db.Uuid
  unitId     String?  @map("unit_id") @db.Uuid
  url        String   @map("url")
  sort_order Int      @default(0) @map("sort_order")
  caption_ar String?  @map("caption_ar")
  caption_en String?  @map("caption_en")
  isActive   Boolean  @default(true) @map("is_active")
  createdAt  DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
}
```

**الأجزاء المطلوبة:**
1. Schema migration — إضافة `GalleryImage`
2. Backend endpoints — CRUD + reorder
3. Admin UI — واجهة drag & drop في `UnitFormModal`
4. Public API — جلب الصور المرتبة للعرض

**الأثر:** متوسط التعقيد — 4 أجزاء.

## 🖼️ SPRINT EXECUTION: ARCHITECTURE PLAN FOR DYNAMIC GALLERY

**Context:** Adopting a UI-First, Design-Driven approach for the new Image Gallery on the public unit page.

### 1. Objective
Build a premium, **Bento-Box style** image gallery for the public unit page. The gallery must be fully dynamic and structured so it can be managed and edited seamlessly from the Admin Dashboard in a Headless CMS style.

### 2. Component Architecture (Props Design)
The `UnitImageGallery` component will expect a structured JSON array rather than a simple string array of URLs. This structure is essential for rendering varying layouts (like a Bento Box) and providing captions.

**Expected Prop Structure:**
```typescript
interface GalleryImage {
  id: string;
  url: string;
  caption_ar?: string;
  caption_en?: string;
  span_size: 'small' | 'medium' | 'large'; // Determines the grid span in the bento layout
  sort_order: number;
}

// Component Prop
interface UnitImageGalleryProps {
  images: GalleryImage[];
}
```

### 3. Styling & Physics Constraints
- **Styling:** Exclusively use **Tailwind CSS**. Implement high-end **Glassmorphism** effects (e.g., `backdrop-blur-md bg-white/10`) for overlays, captions, and UI controls within the gallery.
- **Animations:** All interactions (hover effects, modal opening, expanding images) must be driven by **Framer Motion**.
- **Physics Preset:** Must strictly use the designated "Smooth Spatial" Spring physics preset for a premium feel:
  `{ type: "spring", stiffness: 60, damping: 20 }`

### 4. Admin Integration Roadmap
To enable the Admin to control this gallery:
1. **Database Update:** The `GalleryImage` model (which was identified as missing in the previous audit) will be created in the Prisma Schema to persist the structured data (`url`, `caption`, `span_size`, `sort_order`).
2. **Dashboard UI:** A Drag-and-Drop interface will be built in the Admin Dashboard (inside `UnitFormModal` or a dedicated Gallery tab). Admins will be able to upload images, reorder them, select their display size (`span_size` for the Bento box), and add captions.
3. **API:** The `/public/{slug}/listings` endpoint will be updated to fetch and return the structured `GalleryImage` JSON array instead of just a flat list of string URLs.

### 5. Phase 1 Execution Steps
Before touching the backend or database, the execution will follow the UI-First approach:
1. Scaffold the `UnitImageGallery.jsx` component.
2. Define a **dummy structured data array** that matches the `GalleryImage` interface.
3. Build the Bento-Box CSS Grid layout using Tailwind.
4. Implement the Glassmorphism overlays and Framer Motion spring physics for hover and click interactions.
5. Review the visual result and ensure it meets the Awwwards-level standard before proceeding to Phase 2 (Backend Integration).

### 🔴 CRITICAL — Security Issues (أصلح أولاً)

#### [SEC-01] Plain-text Password Comparison
**الملف:** `app/core/security.py`
- الـ `verify_password` يقبل plain text إذا الـ hash ما يبدأ بـ `$2`
- يسمح بـ timing attack ويكشف الباسوردات لو اخترق الـ DB
- **الإصلاح:** إذا الـ hash مش bcrypt → `return False` ولا مقارنة
- **الحالة:** ⬜ Pending

#### [SEC-02] Race Condition في `create_public_booking`
**الملف:** `app/services/public_service.py`
- لا يوجد availability check قبل `db.booking.create` → شخصان يحجزان نفس الوحدة
- **الإصلاح:** أضف `db.booking.find_first` للتحقق من conflict قبل الإنشاء
- **الحالة:** ⬜ Pending

#### [SEC-03] ملف `.env` في الـ ZIP
- Credentials حقيقية مكشوفة (DATABASE_URL, WHATSAPP_TOKEN, SECRET_KEY)
- **الإصلاح الفوري:** تحقق من `.gitignore`، ثم rotate كل الـ keys في Supabase + Meta
- **الحالة:** ⬜ Pending

#### [SEC-04] Multi-Tenant Phone Uniqueness
**الملف:** `prisma/schema.prisma`
- `Customer.phone @unique` و `email @unique` عالمي → يفشل لو نفس العميل يحجز على tenant آخر
- **الإصلاح:** استبدل بـ `@@unique([clientId, phone])` و `@@unique([clientId, email])`
- **الحالة:** ⬜ Pending
