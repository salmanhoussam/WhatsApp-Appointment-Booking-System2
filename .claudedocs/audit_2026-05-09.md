## Issues Found

### ARCH-01 — Prisma Calls in Route Handlers
**Severity:** 🟠 High (pre-existing systematic pattern)  
**Scope:** 20 route files across `app/api/v1/`

**Files (ordered by call count):**
| File | Direct Prisma Calls |
|------|---------------------|
| `app/api/v1/public/store.py` | 17 |
| `app/api/v1/admin/units.py` | 17 |
| `app/api/v1/admin/restaurant.py` | 16 |
| `app/api/v1/admin/catalog.py` | 16 |
| `app/api/v1/admin/store.py` | 15 |
| `app/api/v1/public/__init__.py` | 12 |
| `app/api/v1/admin/gallery.py` | 11 |
| `app/api/v1/super/platform_services.py` | 9 |
| `app/api/v1/admin/auth.py` | 8 |
| `app/api/v1/public/restaurant.py` | 7 |
| … 10 more files | — |

**Why it's flagged:** CLAUDE.md rule #4 — "لا Prisma calls خارج Repositories". The repository layer is largely bypassed; most modules query Prisma directly from route handlers.

**Why it's not blocking deploy:** This is the established pattern throughout the entire platform — it would require a systematic refactor of all modules. It does not introduce security vulnerabilities (multi-tenant isolation is correctly enforced via `clientId` in every WHERE clause). No data leaks, no runtime crashes.

**Recommended fix path (phased, non-urgent):**
1. Create `app/repositories/` per module (`catalog_repo.py`, `restaurant_repo.py`, …)
2. Extract Prisma calls from routes into repository functions
3. Route → Service → Repository
4. Target: Phase 60+ (post-migration of caracas/footlab)
